# Clean Refactor Implementation Plan

Purpose: convert the observed clean-stage bloat and drift failures into project-agnostic skill fixes. The fixes below should improve any dataset run through `/skeptic:clean`, not only the dataset that exposed the problems.

Scope: skill-side clean specification, clean cycle YAMLs, and the project-side `03_cleaning.py` shape generated by the skill. Project-specific YAML cleanup is included only as a migration target, not as the design center.

Implementation bar:

- Canonical YAML stays compact, structured, and auditable.
- Numeric, categorical, and citation claims are traceable to script evidence or a verified research log.
- User choices happen before irreversible or stability-defining decisions.
- Cycle S can fail for honest instability, not because the agent wrote a passing threshold.
- Generated project scripts reuse shared mechanics instead of reimplementing loaders, dtype checks, and constraint checks per cycle.
- Changes stay project-agnostic: no dataset names, domain-specific thresholds, absolute paths, or one-off column assumptions in the skill.

## Sub-Agent Review Findings

Three parallel reviews compared the fix blocks against the full skill surface. Their findings changed this plan in five ways:

- Cycle F should not be deleted. It is the only explicit clean-stage gate for derived variables and downstream-safe transforms. Keep it conditional, tighten it, and preserve the skip rule.
- Evaluation subagents stay mandatory because `blocking_failures` depends on their unanswered-item, blocking-defect, and failed-gate counts. Only the research subagent becomes conditional.
- `auto-mode.md`, `constraint-spec.md`, and `references/formulate/script_shape.py` contain stale or conflicting mechanics. Fix those before relying on clean-only simplification.
- Citation handling needs `source_ref` support for URLs, project files, local codebooks, and upstream artifacts, not only web URLs.
- Cycle S needs executable user approval paths for thresholds and Layer 2 choices, and its "competing policy per question-critical variable" rule must apply only when a relevant competing policy exists.

## Block A: Remove Duplicate Facts From Canonical YAML

### A1. Replace `upstream_snapshot` With Pointers and Hashes

Status: sound, but requires coordinated schema migration.

Problem: `references/clean/clean.md` defines `upstream_snapshot` as a copy of formulation and protocol facts. Cycle A also initializes `upstream_snapshot`. This duplicates upstream canonical YAML and invites drift.

Project-agnostic change:

```yaml
upstream_refs:
  formulation:
    path: "{docs_dir_name}/01_formulation.yaml"
    sha256:
    fields_used: []
  protocol:
    path: "{docs_dir_name}/02_protocol.yaml"
    sha256:
    fields_used: []
```

Runtime rule: read upstream facts from the upstream YAMLs when needed, verify the hashes before using them, and record only the field names used in the clean stage.

Implementation locations:

- `references/clean/clean.md`: canonical YAML schema, Cycle A setup text, cycle protocol, subagent prompt context.
- `references/clean/cycles/A.yaml`: setup side effects and any checklist language that writes `upstream_snapshot`.
- `references/clean/cycles/C.yaml`, `F.yaml`, `S.yaml`: replace `upstream_snapshot.*` upstream dependencies with `upstream_refs.*` plus explicit reload-from-upstream instructions.
- Route-specific clean files only if they name `upstream_snapshot`.
- `references/core-principles.md`: if `upstream_refs` becomes a cross-stage pattern.

Risk: downstream prompt templates currently assume copied fields are in `03_cleaning.yaml`. The migration must define a helper/read pattern so the model still has the approved question, claim boundary, prohibitions, and data usage mode in context.

Additional drift to fix: `references/clean/cycles/A.yaml` names upstream fields such as `protocol.route` and `protocol.data_usage_mode`. Align them to the actual protocol schema before adding `upstream_refs`, for example `route.active`, `data_usage.mode`, `evidence_rules.*`, `prohibitions.clean`, and frozen-artifact provenance fields.

### A2. Record File Hashes Once

Status: sound and already aligned with `references/core-principles.md`, which says per-file provenance lands once in `provenance.files`.

Project-agnostic change: add `provenance.files` to the clean schema, or explicitly reference `01_formulation.yaml` `provenance.files` through `upstream_refs`. After Cycle A, evidence packets and `cycle_history[*].script_evidence` refer to files by filename.

Implementation locations:

- `references/clean/clean.md`: schema and Step 1/Step 5 rules.
- `references/clean/cycles/A.yaml`: make `precondition_check.hash_verification` a pass/fail summary, not the canonical hash store.
- `references/protocol/protocol.md`: verify protocol-created frozen artifacts have hash fields clean can check.
- Generated script shape: `load_raw()` uses the verified provenance record and encoding.

Risk: if clean does not own `provenance.files`, the spec must state which upstream file owns it and how hash drift is detected.

Additional drift to fix: `clean.md` references `provenance.files`, but `03_cleaning.yaml` currently has no top-level `provenance` field. Decide whether clean gets one or clean always references upstream provenance.

### A3. Delete Top-Level `visibility`

Status: modify before accepting. Visibility is a protocol-owned rule, but clean still needs a compact stage-local access snapshot for audit and prompts.

Project-agnostic change: replace top-level `visibility` with `access_refs` or `visibility_ref`:

```yaml
access_refs:
  source: "{docs_dir_name}/02_protocol.yaml#visibility_rules"
  source_sha256:
  visible_artifacts_used: []
  restricted_artifacts_seen: []
```

The field should not restate all protocol visibility rules. It should record only what clean used and the hash of the protocol source.

Implementation locations:

- `references/clean/clean.md`: schema, Stage Outputs description, Step 1, Step 2/3 prompts, final markdown sections, README block.
- `references/clean/cycles/A.yaml`, `B.yaml`, `C.yaml`, `E.yaml`, `R.yaml`: replace `visibility.*` upstream fields and skip conditions.
- `references/core-principles.md`: only if compact access refs become a global stage pattern.

Risk: fully deleting all visibility material would weaken restricted-artifact auditability. Keep a compact usage record.

### A6. Shrink `pcs_review.verbatim`

Status: sound, with an audit-trail sidecar.

Project-agnostic change:

```yaml
pcs_review:

    predictability:
    stability:
    computability:
  disposition:
  disposition_reason:
  open_conditions: []
  full_review_ref: "{docs_dir_name}/pcs_review.json"
```

Store the full response as JSON or markdown sidecar only when audit depth is needed. The canonical YAML keeps the structured verdict.

Implementation locations:

- `references/clean/clean.md`: PCS schema, PCS review instructions, finalization scorecard.
- Generated project closeout behavior.

Risk: `core-principles.md` says documentation lives in canonical YAML, not side notes. The compromise is that decisions and conditions stay canonical; raw reviewer prose is non-canonical audit evidence.

### A7. Enforce One-Line `script_evidence`

Status: sound and already partly specified.

Project-agnostic change: promote the existing rule into a validation check:

- 4-8 bullets or one value per `evidence_key`.
- No arrays, DataFrames, schemas, hashes, per-column dumps, or prose paragraphs.
- Full details remain in `scripts/stdout/cycle_X.json`.

Implementation locations:

- `references/clean/clean.md`: Step 5 and generated YAML validation rules.
- Future check script: parse `03_cleaning.yaml` and fail if `script_evidence` values exceed a one-line or configured character limit.

Risk: too strict a line-length rule may reject legitimate compact summaries. Prefer structural checks first, line-length warnings second.

## Block B: Stop YAML From Inviting False Narrative

### B1. Require Evidence Keys for Numeric and Categorical Claims

Status: sound, but implement as structured references, not a fragile prose scanner alone.

Project-agnostic change: canonical YAML prose fields may not introduce numeric or categorical claims unless they include `evidence_key` or `evidence_ref`.

Example:

```yaml
population_shift:
  evidence_ref: cycle_C.row_count_and_population
  interpretation: "Removes records outside the approved analyzable population."
```

Checker behavior:

- Error when a YAML scalar outside script evidence contains digits, percentages, comparison operators, set notation, or categorical enumerations without a nearby `evidence_ref`.
- Allowlist timestamps, schema versions, checklist IDs, cycle IDs, hashes, file paths, and row-count fields designed to hold numbers.

Implementation locations:

- `references/clean/clean.md`: new canonical YAML rule.
- `references/clean/cycles/C.yaml`, `S.yaml`: require evidence refs on population shifts and threshold fields.
- New skill-side or project-side validation helper, likely under `references/clean/` or described in script shape.

Risk: pure regex scanning will false-positive on IDs and dates. The checker should parse YAML and apply field-aware rules.

### B2. Move Research Citations Into `research_log.jsonl`

Status: sound, but preserve canonical decision rationale.

Project-agnostic change: research findings are append-only JSONL records. YAML keeps stable references.

```json
{"id":"research_log#7","cycle":"C","url":"https://...","claim_used":"...","verified_at":"...","used_by":["cleaning_decisions[3]"]}
```

YAML example:

```yaml
rationale_source:
  type: research_log
  ref: research_log#7
```

Implementation locations:

- `references/clean/clean.md`: Stage Outputs optional companion artifacts, `cycle_history.subagents.research_sources`, Step 3 digest rules.
- All clean cycle YAMLs: no need to change all research questions, but update guidance so research output is logged externally and referenced canonically.
- Generated project artifact: `{docs_dir_name}/research_log.jsonl` or `{scripts_dir_name}/research_log.jsonl`; prefer docs because it is an audit artifact.
- All stage entry files if `source_ref` becomes global.

Risk: citations in a sidecar can drift from decisions if refs are not validated. Add a check that every `research_log#N` used in YAML exists and every log row has a URL, claim, cycle, and verification timestamp.

Modification from review: use a general `source_ref` schema rather than a web-only citation schema:

```yaml
source_ref:
  id: source#7
  kind: url|project_file|upstream_artifact|codebook
  locator:
  claim_used:
  verified_at:
```

If `cycle_history.subagents.decisions[*].source` remains an integer, define whether it is zero-based or one-based. Prefer stable IDs instead.

## Block C: Ask the User at the Right Moments

### C1. Mandatory Check-Ins at Cycles B, C, and S

Status: sound for interactive mode; auto mode needs a parallel recorded-decision path.

Project-agnostic change:

- Cycle B: ask one question summarizing the highest-impact missing-vs-censored or integrity-policy ambiguity.
- Cycle C: ask one question before irreversible exclusions or transformations affecting more than a configured materiality threshold.
- Cycle S: ask one question approving derived or externally sourced instability thresholds.
- Cycle S: ask the Layer 2 choice when Layer 2 is plausible but optional.
- Dataset fitness: surface any `no` or `unclear` answer as a decision point.
- Stage close: keep PCS disposition and final clean-stage approval as explicit interactive gates.

Implementation locations:

- `references/clean/clean.md`: Step 2 Human Review and auto-mode exception.
- `references/clean/cycles/B.yaml`: add `user_checkin` metadata for missing/censored policy ambiguity.
- `references/clean/cycles/C.yaml`: add `user_checkin` metadata for irreversible actions and material population shifts.
- `references/clean/cycles/S.yaml`: add `user_checkin` metadata for threshold approval.

Risk: hard-coding `0.1%` as universal materiality is not project-agnostic. Use a default materiality rule that can be overridden by protocol or user preference:

```yaml
materiality:
  default_row_fraction: 0.001
  override_source: protocol_or_user
```

Do not turn every cycle into a user approval gate. The goal is to force user input at the decisions the model cannot legitimately self-approve.

## Block D: Make Cycle S Actually Test Stability

### D1. Derive Instability Thresholds

Status: sound, but threshold derivation must be route- and metric-aware.

Project-agnostic change: Cycle S must produce thresholds from one of:

- protocol-defined or external standard;
- bootstrap or resampling variability of the metric;
- documented measurement resolution or domain tolerance;
- user-approved threshold when no defensible automatic source exists.

The agent may propose a threshold source, but cannot silently invent the passing number.

Implementation locations:

- `references/clean/cycles/S.yaml`: S01 evidence shape and gate.
- `references/clean/clean.md`: Cycle S script-shape instructions and PCS review prompt.
- Generated `run_cycle_s`: threshold derivation helper and user approval state.

Risk: `2 x bootstrap std` is a useful example, not a universal default. The spec should require a threshold-source hierarchy, not one formula.

Additional requirement: S01 currently requires approval but does not guarantee an approval path. Add explicit interactive and auto-mode paths so `approval_state` is executable, not aspirational.

### D2. Derive Expected Exceedances From Metric Dependencies

Status: sound, with a bounded implementation.

Project-agnostic change: `run_cycle_s` builds a dependency graph from declared metric formulas and judgment-call parameters. Expected moved metrics are derived from graph reachability, not hand-written maps.

Implementation locations:

- `references/clean/cycles/S.yaml`: add evidence for dependency graph and expected affected metrics.
- `references/clean/clean.md`: generated script shape for Cycle S.
- Generated project script: formulas for question-critical variables and derived variables must be inspectable enough to build the graph.

Risk: arbitrary pandas code is hard to statically analyze. Require project scripts to register metric dependencies explicitly when formulas cannot be inferred:

```python
metric_dependencies = {
    "metric_name": {"columns": [...], "judgment_params": [...]}
}
```

Modification from review: the "one competing policy per question-critical variable" rule should apply only when a meaningful competing policy exists. Otherwise `run_cycle_s` records `no_relevant_competing_policy` with a reason.

## Block E: Fix Broken Project-Side Mechanics

### E1. Simplify Constraint Verification

Status: sound.

Project-agnostic change: generated verifiers use:

```python
bad = ser[ser.notna() & ~ser.isin(allowed)]
```

Null admissibility is checked separately through constraint metadata:

```json
{"nullable": true}
```

Implementation locations:

- `references/constraint-spec.md`: ensure nullable semantics are explicit.
- `references/clean/clean.md`: Cycle R helper and constraint-check rules.
- Generated `03_cleaning.py`: `_verify_clean_constraints` and `_verify_preprocess_constraints`.
- `references/auto-mode.md`: remove stale notebook-based constraint checks before enforcing new script checks.

Risk: some domains distinguish `NaN`, empty string, and sentinel missing codes. That distinction belongs in earlier cleaning decisions and constraints, not in an ad hoc null branch.

### E2. Load Raw Data Once

Status: sound.

Project-agnostic change: generated `03_cleaning.py` includes memoized loaders:

```python
@lru_cache(maxsize=None)
def load_raw(filename: str) -> pd.DataFrame:
    ...
```

All cycles call the same loader, with encoding and dtype policy sourced from provenance/schema metadata.

Implementation locations:

- `references/clean/clean.md`: script shape.
- `references/clean/cycles/A.yaml`: provenance and dtype capture.
- `references/formulate/script_shape.py`: if helper-surface rules are shared from the base script template.
- Generated `03_cleaning.py`: all `run_cycle_*` functions.

Risk: memoization is per process. Because cycles are invoked as separate CLI runs, this mainly unifies code and dtype policy, not cross-cycle memory. That is still the important fix.

### E3. Centralize Dtype Meaning Checks

Status: sound.

Project-agnostic change: generated script includes `check_dtype_meaning(series, expected_meaning)` or a small equivalent helper that classifies semantic dtype families rather than exact pandas dtype strings.

Implementation locations:

- `references/clean/clean.md`: script shape and Cycle A guidance.
- `references/clean/cycles/A.yaml`: A02 evidence shape and gate.
- Generated `03_cleaning.py`: helper used by all cycles that compare dtypes.

Risk: the helper must be semantic, not a growing list of pandas dtype spellings. Use categories such as string-like, integer-like, decimal-like, datetime-like, boolean-like, categorical-like, identifier-like.

## Block F: Trim the Cycle Spec Itself

### F1. Merge Cycle E Into Cycle C Closeout

Status: promising but highest schema risk. Treat as conditionalization first, deletion second.

Project-agnostic change: convert preprocessing from a mandatory cycle into a conditional closeout section after cleaning resolution:

- keep fit-scope leakage checks;
- keep grouped retained-as-is decisions;
- defer route/model-shaped transforms downstream;
- open a G-series follow-up only when representation work is material and cannot be closed in C.

Implementation locations:

- `references/clean/clean.md`: Stage Outputs, Cycle Structure, Ending the Cycle Loop, Cycle R dependencies, finalization scorecard.
- Delete or deprecate `references/clean/cycles/E.yaml`.
- `references/clean/cycles/C.yaml`: add closeout fields for low-regret representation decisions and fit-scope audit.
- `references/clean/cycles/R.yaml`: remove assumptions that E always ran.
- `references/clean/cycles/G_template.yaml`: update trigger language.

Risk: preprocessing can be real work for some datasets. Do not delete the concept. Make it conditional and issue-centered rather than mandatory.

Sub-agent constraint: do not delete Cycle F as part of this work. Cycle F is the derived-variable gate and remains conditional.

### F2. Delete Pure Bookkeeping Checklist Items

### F3. Scale Back Dual Research and Evaluation Subagents

Status: sound with a trigger-based escape hatch.

Project-agnostic change:

- Always run evaluation subagent for each cycle.
- Run research subagent by default only on B, C, and S.
- Allow research on A, F, R, or follow-up cycles when a cycle-specific script result surfaces a domain ambiguity that can change a decision.

Implementation locations:

- `references/clean/clean.md`: Step 3 Subagent Review.
- Each clean cycle YAML: add `research_policy: default_on|conditional`.
- `references/core-principles.md` only if this policy should generalize beyond clean.

Risk: Cycle A sometimes needs domain research for field semantics. The policy should be conditional, not a hard ban.

Non-negotiable: evaluation subagents remain mandatory unless the global decision matrix changes. `blocking_failures` relies on their counts.

- `row_count_reconciliation`.

## Recommended Implementation Order

1. Global contract cleanup: fix `core-principles.md`, `auto-mode.md`, `constraint-spec.md`, and any stale notebook/metrics/claim-registry references that conflict with the current artifact model.
2. Mechanics: E1, E2, E3.
3. Local YAML compaction patterns: A4, A5, A6, A7.
4. Evidence discipline: B1, B2, including `source_ref`.
5. Upstream and visibility references: A1, A2, A3.
6. Checklist and subagent pruning: F2, F3, while keeping evaluation mandatory and Cycle F conditional.
7. Mandatory user check-ins: C1.
8. Conditionalize Cycle E; delete it only after the conditional path is stable.
9. Cycle S stability redesign: D1, D2.

Reasoning: first remove cross-file contract contradictions, then fix generated-script bugs, then reduce YAML volume, then change schema and workflow, then tackle the hardest stability logic.

## Acceptance Tests

- A generated `03_cleaning.yaml` has no copied upstream formulation/protocol prose; it has upstream refs and hashes.
- A raw file hash appears in exactly one canonical provenance location.
- Clean-stage access records identify artifacts used without copying the full protocol visibility rules.
- Cycle C outputs one issue-centered list that still covers options, chosen policy, code ref, population shift, and claim consequence.
- Repeated retained-as-is preprocessing decisions can be grouped when they share rationale.
- PCS review in YAML contains structured lens verdicts and disposition, with optional sidecar for full text.
- Any numeric or categorical prose claim in canonical YAML carries `evidence_ref`, except allowlisted structural fields.
- Any YAML research citation ref resolves to a row in `research_log.jsonl`.
- Interactive Cycle B, C, and S runs ask the mandatory user question or record an auto-mode equivalent.
- Cycle S thresholds have a source and approval state. No threshold is agent-invented without source.
- Cycle S expected affected metrics are derived from declared metric dependencies.
- Constraint verification treats nulls through `nullable`, not special-case branches.
- All generated cycle functions read raw data through the same loader and dtype policy.
- Removing bookkeeping items does not remove the underlying enforceable guarantee.

## Open Decisions

- Whether `upstream_refs` should store upstream file hashes only, or also per-field hashes.
- Whether `access_refs` belongs in canonical YAML or can be fully derived from protocol plus script stdout.
- Whether `research_log.jsonl` should live in `{docs_dir_name}` or `{scripts_dir_name}`. Current recommendation: `{docs_dir_name}`.
- Whether Cycle E is deprecated immediately or first made conditional for one release.
- How strict the numeric-claim checker should be on first rollout: warning-only first, then blocking after migration is easier.
